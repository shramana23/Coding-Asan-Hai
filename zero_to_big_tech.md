## 🧠 Before You Start — The Mindset

Before touching a single line of code, internalise these:

**1. This is a 12-month journey, not a 12-week sprint.**
Anyone telling you they cracked Google in 6 weeks either had years of background already or is lying. Set a realistic timeline and protect it.

**2. Confusion is not a sign you're failing. It's a sign you're learning.**
Every engineer — including ones at Google — Googles basic syntax. The discomfort is the progress.

**3. Consistency beats intensity.**
2 hours every day for 12 months is worth more than 12-hour sessions on weekends. The brain consolidates learning during rest, not during cramming.

**4. You are not competing against everyone. You are competing against your yesterday.**
The goal is not to be better than others right now. It is to be better than you were last week, every week.

**5. Build things. Reading and watching is not learning.**
You do not learn swimming by watching swimming videos. You learn by getting in the water. Code every day. Even 20 lines. Even if it breaks.

| **C++** | DSA, competitive programming, systems | Fastest for interviews, powerful |

## 🟠 Phase 2 — DSA Foundations (Weeks 11–22)

> **Goal:** Build the problem-solving brain that big tech companies test for. Learn patterns, not just solutions.

### The Most Important Thing About DSA

You are not memorising solutions. You are learning **patterns**. Every LeetCode problem is a variation of one of ~15 core patterns. Once you see the pattern, you can solve any variation.

The patterns are:
```
1. Two Pointers
2. Sliding Window
3. Fast & Slow Pointers
4. Merge Intervals
5. Cyclic Sort
6. In-place Reversal of Linked List
7. Tree BFS (Level Order Traversal)
8. Tree DFS
9. Two Heaps
10. Subsets / Backtracking
11. Binary Search
12. Top K Elements (Heap)
13. K-Way Merge
14. Dynamic Programming
15. Topological Sort (Graphs)
```

Learn these patterns. Everything else is a variation.

### DSA Study Method (Do This Every Day)

```
Step 1 — Read the problem. Don't look at hints. Think for 15 minutes.
Step 2 — If stuck: look at the hint only (not the solution).
Step 3 — If still stuck after 30 min total: read the solution.
Step 4 — Understand the solution completely. Can you explain it?
Step 5 — Close everything. Code it from scratch.
Step 6 — 24 hours later: solve the same problem again without looking.
```

This spaced repetition approach is what separates people who remember patterns from people who recognise them in hindsight.

## 🔴 Phase 4 — Advanced DSA + System Design (Weeks 35–46)

> **Goal:** Clear the hardest coding rounds. Design systems that handle millions of users.

### Advanced DSA (Weeks 35–40)

#### Dynamic Programming
The most feared topic. The most pattern-driven topic. Once it clicks, it stays.

```
Start with: Fibonacci, Climbing Stairs, House Robber
Then: 0/1 Knapsack, Unbounded Knapsack
Then: LCS, LIS, Edit Distance
Then: Matrix DP, Interval DP
```

The DP thought process:
```
1. Define the subproblem — what does dp[i] mean?
2. Find the recurrence — how does dp[i] relate to dp[i-1]?
3. Identify base cases
4. Decide: top-down (memoisation) or bottom-up (tabulation)?
5. Optimise space if possible
```

Must-solve DP problems:
- Climbing Stairs (#70)
- House Robber (#198)
- Longest Common Subsequence (#1143)
- Coin Change (#322)
- Word Break (#139)
- Partition Equal Subset Sum (#416)
- Edit Distance (#72)

---

#### Backtracking
```
Pattern: build → check → undo
Use when: you need all combinations/permutations/subsets
```

Must-solve:

### System Design (Weeks 41–46)

System Design is tested for SDE-2 and above at most companies. For freshers and new grads, knowing the basics is a strong differentiator.

#### The Framework — Use This Every Time

```
Step 1 — Clarify requirements (5 mins)
  → Functional: what does it do?
  → Non-functional: scale, latency, availability

Step 2 — Estimate scale (5 mins)
  → Daily Active Users (DAU)
  → Requests per second
  → Storage needed

Step 3 — High-level design (10 mins)
  → Draw the main components
  → Client → Load Balancer → App Servers → Database → Cache

Step 4 — Deep dive into components (20 mins)
  → Database design — schema, SQL vs NoSQL choice
  → Caching strategy — what to cache, TTL, eviction
  → API design — endpoints, request/response

Step 5 — Identify and solve bottlenecks (10 mins)
  → Single points of failure
  → Scaling the database (sharding, replication)
  → CDN for static assets
```

#### Core Concepts to Master

```
📦 Databases
  → SQL (PostgreSQL) vs NoSQL (MongoDB, DynamoDB, Cassandra)
  → When to use each, tradeoffs
  → Indexing — how it works, when it helps, when it hurts
  → Sharding, replication, consistency vs availability (CAP theorem)

⚡ Caching
  → Redis, Memcached
  → Cache-aside, write-through, write-back patterns
  → Cache invalidation — the hardest problem in CS

🌐 APIs
  → REST vs GraphQL vs gRPC
  → Rate limiting, throttling, pagination
  → Idempotency

📨 Message Queues
  → Kafka, RabbitMQ, SQS
  → When to use async vs sync
  → Fan-out, pub-sub patterns

🔄 Load Balancing
  → Round robin, least connections, consistent hashing
  → L4 vs L7 load balancers

🔒 Security basics
  → HTTPS, TLS
  → Authentication vs Authorisation
  → SQL injection, XSS — what they are and how to prevent
```

#### System Design Problems to Practice

| Problem | Key Concepts Tested |
|---|---|
| Design a URL Shortener | Hashing, DB design, caching, redirects |
| Design Twitter / Instagram Feed | Fan-out, feeds, CDN, sharding |
| Design WhatsApp | WebSockets, message queue, end-to-end encryption |
| Design YouTube | Video storage, CDN, encoding pipeline, streaming |
| Design Google Search | Crawling, indexing, ranking, distributed search |
| Design Uber | Geospatial indexing, real-time location, matching |
| Design a Rate Limiter | Token bucket, sliding window, distributed counters |
| Design Notification System | Push vs pull, message queues, fan-out |

## 🎯 Phase 5 — Interview Preparation (Weeks 47–52)

> **Goal:** Convert your preparation into an offer. Practice is different from preparation.

### The Interview Is a Skill. Train It.

Most people prepare for interviews by doing more LeetCode. That's necessary but not sufficient. The interview also tests:
- How you think out loud
- How you handle hints from the interviewer
- How you handle a problem you've never seen
- How you communicate under pressure

**These are trainable skills. Train them.**

---

### Coding Interview Practice

**Daily routine for Phase 5:**
```
Morning (45 mins): 1 new problem — attempt without hints
Evening (45 mins): 1 mock interview with a friend or on Pramp
```

**Mock interview platforms:**
- [Pramp](https://www.pramp.com) — free, peer mock interviews, highly recommended
- [Interviewing.io](https://interviewing.io) — mock interviews with real engineers, anonymous
- [LeetCode Mock](https://leetcode.com/interview/) — timed, company-specific
- With friends — best option, talk out loud, give each other feedback

**The UMPIRE method for solving problems in interviews:**
```
U — Understand: repeat the problem back, ask clarifying questions
M — Match: what pattern does this look like?
P — Plan: talk through your approach before coding
I — Implement: code it, narrate as you go
R — Review: check for bugs, edge cases
E — Evaluate: what's the time and space complexity?
```

---

### Behavioural Interview Preparation

Every big tech company has a behavioural round. Google calls it "Googleyness." Amazon calls it "Leadership Principles." It is not soft. It is structured. Prepare for it like a technical round.

**Write 8-10 STAR stories covering these themes:**

| Theme | Example Question |
|---|---|
| Leadership | Tell me about a time you led a project without formal authority |
| Conflict | Tell me about a time you disagreed with a teammate or manager |
| Failure | Tell me about your biggest failure. What did you learn? |
| Impact | Tell me about the most impactful thing you've built |
| Ambiguity | Tell me about a time you had to make a decision with incomplete information |
| Collaboration | Tell me about a time you helped a teammate succeed |
| Growth | Tell me about a time you learned something difficult |
| Innovation | Tell me about a time you found a creative solution to a problem |

**The STAR format:**
```
Situation — set the context (2-3 sentences)
Task       — what was your responsibility (1-2 sentences)
Action     — what YOU specifically did (4-5 sentences — this is the meat)
Result     — what happened, with numbers if possible (2-3 sentences)
```

## 🏢 The Big Tech Interview — What Actually Happens

Here is the exact process at each stage so there are no surprises.

### Stage 1 — Resume Screen
A recruiter reviews your resume. They look for: target school/company tier, relevant experience, and keywords matching the role. This is why your resume must be ATS-optimised with the right keywords (Python, REST API, System Design, etc.)

**Timeline:** 1-3 weeks after applying

---

### Stage 2 — Online Assessment (OA)
2-3 coding problems on HackerRank, CodeSignal, or similar. Timed: usually 60-90 minutes. Difficulty: Easy to Medium. No interviewer. Just you and a timer.

**Tip:** OAs at big companies have a pass bar — usually 2/3 problems fully solved. Focus on correctness first, then speed.

**Timeline:** 1-2 weeks after resume screen

---

### Stage 3 — Recruiter Call
15-30 minute call. Non-technical. They ask: tell me about yourself, why this company, what are you looking for.

**What to prepare:** A crisp 90-second "about me" story. Know the company's products and recent news. Have 2-3 questions ready for them.

**Timeline:** 1 week after OA results

---

### Stage 4 — Technical Phone Screen
45-60 minutes with an engineer. 1-2 coding problems. Live coding in a shared editor (CoderPad, Google Docs, or similar).

**What they're watching for:** How you approach the problem, how you communicate, how you handle edge cases and feedback.

**Timeline:** 1-2 weeks after recruiter call

---

### Stage 5 — Onsite / Virtual Onsite
The final round. Usually 4-5 interviews in one day.

| Round | What's tested |
|---|---|
| Coding Round 1 | DSA — Medium to Hard |
| Coding Round 2 | DSA — Medium to Hard |
| System Design | Design a large-scale system |
| Behavioural | STAR stories, culture fit |
| Hiring Manager | Your goals, team fit, past work |

**Timeline:** 1-3 weeks after phone screen

---

### Stage 6 — Offer, Negotiation, Decision
The offer comes via the recruiter. You have time to negotiate — and you should. Data points: use Glassdoor, Levels.fyi, and competing offers.

**Never accept on the call.** Always say: "Thank you so much — I'm really excited. Can I have a few days to review it with my family?" This is professional, expected, and buys you time to negotiate.

## 🏆 Company-Specific Strategies

### Google
```
Strength of signal: highest bar in the industry
Focus: Clean, correct code. Think out loud. Communication is weighted.
Unique round: Googleyness (culture fit — prepare STAR stories)
Process: OA → 2 phone screens → 5 virtual onsites
Tip: Google cares deeply about how you explain your reasoning.
     A slightly slower, clearly communicated solution beats
     a fast solution with no explanation.
```

### Microsoft
```
Process: OA → 1 phone screen → 4-5 onsites
Focus: DSA + behavioural, some system design for senior roles
Tip: Microsoft heavily tests communication and mentorship mindset.
     "How would you explain this to a junior engineer?" is a common follow-up.
```

### Amazon
```
Process: OA → phone screen → 5-6 onsites (including 1 Bar Raiser)
Focus: 14 Leadership Principles — these are baked into EVERY round
Tip: You will be asked LP questions in every single round, not just behavioural.
     Know all 14 principles. Have a story for each one.
     The Bar Raiser round is the hardest — it's run by a senior employee
     from a different team whose job is to raise the overall hiring bar.
```

### Meta / Flipkart / Swiggy / Other Product Companies
```
Process: similar to above, typically 4-5 rounds
Focus: product sense is valued more here than at pure engineering shops
Tip: know the product deeply. Have opinions on what could be better.
     "What would you change about our app?" is a real interview question here.
```

## ❌ Mistakes That Kill Momentum

| Mistake | Why it hurts | What to do instead |
|---|---|---|
| Tutorial hell | You feel productive but build nothing | Follow any tutorial once, then build without it |
| Random LeetCode grinding | No pattern recognition develops | Follow a structured sheet by topic |
| Solving and moving on | You'll forget in 3 days | Revisit every problem after 24 hours and 1 week |
| Skipping the basics | Your foundation cracks under interview pressure | Do Phase 0 and 1 properly, even if it feels slow |
| Only doing Easy problems | Medium is where interviews live | Move to Medium by Week 12 at the latest |
| Preparing alone | No feedback loop | Find 1-2 prep partners |
| Waiting to apply | The process takes weeks — start early | Apply while you're still preparing |
| One-company focus | Too much riding on one result | Apply to 20-30 companies in parallel |
| Giving up after one rejection | Everyone gets rejected, including great engineers | Request feedback, fix the gap, reapply in 6 months |

## 📅 The Weekly Schedule That Works

**If you're a student (studying full time):**
```
Monday      2 hours DSA (2 problems, topic-focused)
Tuesday     2 hours Development (build your project)
Wednesday   2 hours DSA (2 problems + review yesterday's)
Thursday    2 hours Development
Friday      1.5 hours DSA + 30 min review week's problems
Saturday    3 hours — longer project session or system design study
Sunday      Rest OR 1 hour light reading (newsletter, article, YouTube)
```

**If you're working full time:**
```
Weekday mornings   45 mins DSA (1 problem before work)
Weekday evenings   45 mins development or review
Saturday           3 hours focused (1.5 DSA + 1.5 development)
Sunday             2 hours system design or mock interview
```

**Non-negotiable rules:**
- Code every single day. Even 20 minutes. Consistency is everything.
- No social media during your study block. Phone in another room.
- Track what you study. A simple text file with date + what you did.
- Review your weak areas on Sundays. Don't just practice what you're good at.

## ❓ FAQs — Honest Answers

**Q: Do I need a CS degree?**
A: No. A CS degree gives you structured learning and campus placements. It doesn't guarantee a job. Engineers without degrees work at every big tech company. Your skills and ability to demonstrate them matter more.

**Q: What if my college CGPA is low?**
A: Matters less than you think at most companies. Some companies have a CGPA cutoff for campus hiring (usually 6.5-7.5). Off-campus applications have no CGPA filter — you're judged entirely on your skills.

**Q: How many LeetCode problems do I need?**
A: 250-350 well-understood problems beat 1000 half-understood ones. Quality of understanding over quantity of attempted problems.

**Q: Should I do competitive programming?**
A: It helps with speed and problem-solving instincts. But CP (Codeforces, CodeChef) and interview prep (LeetCode) are different skills. Don't substitute one for the other. If you enjoy CP, do it — but add LeetCode on top.

**Q: I failed an interview. What now?**
A: Every engineer has a rejection story. Some of the best engineers I know failed their first Google attempt. Request feedback from the recruiter (most will give it). Identify the gap. Give yourself 1-2 weeks to process, then get back to the plan. You can reapply to most companies after 6-12 months.

**Q: Is it too late to start at [age/year]?**
A: No. There are engineers who made it to big tech at 28, 30, 35. The path takes longer if you start later — but the path exists. What matters is that you start, and that you're consistent.

**Q: Can I skip system design if I'm a fresher?**
A: For most fresher roles, yes — system design is not expected. But knowing the basics (enough to have a conversation) is a differentiator. If you have time, do Phase 4's system design intro. If you don't, skip it and focus on DSA and projects.
